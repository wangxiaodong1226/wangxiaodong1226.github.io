### 1.对哈希表理解，解决什么问题?
哈希表是基于数组实现的，可以快速的插入-删除-查找操作。<br>
无论多少数据，插入删除只需接近常量的时间，即O(1)的时间复杂度，只需几个机器指令就可以完成，所以它的查找速度非常快。<br>
哈希表能够通过哈希函数把字符串转化为相应的下标值，建立字符串和下标值的映射关系。<br>
比如说，公司想要存储1000个人的信息，每个工号对应一个员工的信息。如果使用数组的话，增删数据比较麻烦获取数据;使用链表的话，获取数据也比较麻烦；那么就有了哈希表这个<br>数据结构，把某一个员工的姓名转换为它对应的工号，再根据工号查询该员工的完整信息。<br>
* 造成冲突解决办法？
实际中，经过哈希函数哈希化得到的下标值可能有重复，即冲突。<br>
解决冲突有链地址法和开放地址法。<br>
* 解释一下链地址法和开放地址法?
链地址法就是数组每一个下标值对应的位置存储的不再是单个数据，而是一个链条，链条使用的数据结构为数组或链表;<br>
开放地址法就是寻找空白的单元格来放置冲突的数据项；<br>
探测空白单元格位置方式又分为线性探测、二次探测、再哈希法----对步长进行优化。<br>
再哈希法最好，把关键字用另一个哈希函数再做一次哈希化。<br>
stepSize=contant-(key%constant)<br>
* 探测长度的时间跟什么有关？
取决于装填因子，随着装填因子变大，探测长度越来越长。<br>
装填因子为哈希表中已经包含的数据项和整个哈希表长度的比值<br>
* 为什么默认加载因子选择0.75？
装载因子太小容易造成空间浪费，太大容易造成冲突 选择0.75主要是时间和空间上的一种折中选择，泊松分布，0.75碰撞最小。<br>

### 2.进程和线程的区别？
> 我们使用打开一个微信软件，这时候就开启了一个进程，当我们在微信群里面进行各种操作(查看朋友圈，扫一扫)，这么多操作就是线程，所以说‘进程’是包含‘线程的’，‘线程’是‘进程’的一个子集。
进程是并发执行的程序在执行过程中分配和管理资源的基本单位，是一个动态概念，竞争计算机系统资源的基本单位。<br>
线程，是进程的一部分，一个进程至少有一个线程，一个进程可以运行多个线程，多个线程可共享数据，一个没有线程的进程可以被看做是单线程的，线程有时又被称为轻量级进程<br>
* 怎么共享资源？
资源共享就是多个线程对该资源只保存一份。<br>
Java里实现资源共享一个实例就是100章票4个售票点，实现Runnable接口覆盖run方法，把覆盖Runnable接口的类创建一份实例作为参数传递给Thread，把资源保存在Runnable接口的实例里，然后创建一份实现Runnable接口类的实例传递给共享资源的线程就可以了。<br>
* 一个线程挂了整个进程就挂了?
进程（主线程）创建了多个线程，多个子线程均拥有自己独立的栈空间.如果子线程的崩溃是由于自己的一亩三分地引起的，那就不会对主线程和其他子线程产生影响，但是如果子线程的崩溃是因为对共享区域造成了破坏，那么大家就一起崩溃了.<br>

### 3.http协议1.0/1.1/2.0  3.0  解决一些什么样的问题？
http1.0浏览器与服务器只保持短暂的连接，浏览器每次请求都需要与服务器建立一次TCP连接，服务器完成请求处理后立即断开TCP连接,服务器不跟踪每个客户也不记录过去的请求。<br>

http1.1 为了克服HTTP1.0的这个缺陷，HTTP1.1支持持久连接，在一个TCP连接上可以传送多个HTTP请求和响应，减少了建立和关闭连接的消耗和延迟。<br>
HTTP1.1 还允许客户端不用等待上一次请求结果返回，就可以发出下一次请求，但服务器必须按照接收客户端请求的先后顺序一次回送响应结果，以保证客户端能够区分出每次请求的响应内容，这样也显著地减少了整个下载过程所需要的时间。<br>

例如：Collection请求头的值为Keep-Alive时，客户端通知服务器端返回本次请求结果后保持连接，无需再次握手，提高传输速率；Collection请求头为close时，客户端通知服务器返回本次请求结果后关闭连接。<br>
* keep-alive会永久保持连接吗?
不会永久保持连接，它有一个保持时间，可以在不同的服务器软件中设定这个时间，实现长连接需要客户端和服务器端都支持长连接。<br>
HTTP2.0 引入了多路复用，在HTTP1中，我们经常会引入雪碧图等提高性能，因为浏览器限制同一域名下的请求数量，当页面中需要请求很多资源的时候，队头阻塞会导致在达到最大请求数量时，剩余的资源需要等待其他资源请求完后才能发起请求。<br>
于是HTTP2多路复用可以只通过一个TCP连接就可以传输所有的请求数据。能够很好的解决浏览器限制同一域名下的请求数量的问题，同时也更容易实现全速传输，毕竟新开一个TCP连接需要慢慢传输速度。<br>

新的二进制格式 HTTP1.1的解析是基于文本，2.0的协议解析是基于二进制格式，方便且快捷。HTTP/2在 应用层和传输层之间增加一个二进制分帧层，在二进制分帧层中，HTTP/2 会将所有传输的信息分割为更小的消息和帧，并对它们采用二进制格式的编码 <br>

header头部压缩，HTTP1中使用文本的形式传输header，在header携带cookie的情况下，可能每次都需要传输几百到几千的字节。HTTP2对传输的header进行编码，减少了传输的header大小，并且可以维护一个字典，记录出现过的header，差量更新HTTP头部，大大降低因头部传输产生的流量。<br>

服务端推送 服务端推送把客户端所需要的资源伴随着index.html一起发送到客户端，省去了客户端重复请求的步骤。<br>





http3.0
HTTP/2仍然存在一些缺陷，这些缺陷并不是来自于HTTP/2协议本身，而是来源于底层的TCP协议，我们知道TCP链接是可靠的连接，如果出现了丢包，那么整个连接都要等待重传，HTTP/1.1可以同时使用6个TCP连接，一个阻塞另外五个还能工作，但HTTP/2只有一个TCP连接，阻塞的问题便被放大了。

由于TCP协议已经被广泛使用，我们很难直接修改TCP协议，基于此，HTTP/3选择了一个折衷的方法——UDP协议，HTTP/2在UDP的基础上实现多路复用、0-RTT、TLS加密、流量控制、丢包重传等功能
### 4.event loop理解？
JS分为同步任务和异步任务<br>
1.所有同步任务都在主线程上执行，形成一个技术栈<br>
2.主线程之外，还有一个任务队列，只要异步任务有了运行结果，就在任务队列之中放置一个事件<br>
3.一旦执行栈中所有同步任务执行完毕(即JS引擎空闲)，系统就会读取任务队列。将可运行的异步任务添加到可执行栈中，开始执行<br>
4.主线程不断重复<br>
5.从整体代码script开始读取，然后不断循环，从任务队列中读取执行事件的过程就是事件循环<br>

### 5.说一下微任务，宏任务？
宏任务就是每次在执行栈中执行的代码就是一个宏任务，包括每次从事件队列中获取一个事件回调放到执行栈中执行的。<br>
包括script(整体代码) setTimeout setInterval等<br>

微任务当前task执行结束后立即执行的任务<br>
包括Promise.then(回调) 以Promise为基础开发的其他技术 fetch process.nextTick(Node.js环境)<br>

在上一个宏任务队列执行完毕后 如果有微任务队列就会执行微任务队列所有微任务<br>
当宏任务队列中的任务执行完毕后 查看是否有微任务 再执行宏任务<br>

* 手写代码宏任务 微任务
```
    console.log('script start')

    setTimeout(()=>{
        console.log('setTimeout')
    },0)

    Promise.resolve().then(()=>{
        console.log('promise1')
    }).then(()=>{
        console.log('promise2')
    })

    console.log('script end')

```
先执行同步代码，先输出script start->script end.然后调用promise1,将then(promise2)放入微任务，再次调用微任务，将promise2输出。最后调用宏任务setTimeout,输出setTimeout<br>
输出顺序 script start->script end->promise1->promise2->setTimeout<br>

### 8.node的事件循环机制？
node中也有宏任务和微任务，与浏览器的事件循环类似；<br>
宏任务包括setTimeout、setInterval、script、I/O操作等<br>
微任务包括process.nextTick(与普通微任务有区别，在微任务队列执行之前执行)、new Promise().then(回调)等<br>

浏览器的微任务是在每个相应的宏任务中执行的，<br>
而nodejs中的微任务是在不同阶段执行的。<br>

输入数据阶段(incoming data)->轮询阶段(poll)->检查阶段(check)->关闭事件回调阶段(close callback)->定时器检测阶段(timers)->I/O事件回调阶段(I/O callbacks)->闲置阶段(idle, prepare)->轮询阶段..<br>

process.nextTick 是一个独立于 eventLoop 的任务队列。<br>
在每一个 eventLoop 阶段完成后会去检查 nextTick 队列，如果里面有任务，会让这部分任务优先于微任务执行。<br>
```
    setImmediate(() => {
        console.log('timeout1')
        Promise.resolve().then(() => console.log('promise resolve'))
        process.nextTick(() => console.log('next tick1'))
    });
    setImmediate(() => {
        console.log('timeout2')
        process.nextTick(() => console.log('next tick2'))
    });
    setImmediate(() => console.log('timeout3'));
    setImmediate(() => console.log('timeout4'));

```
timeout1->next tick1->promise resolve->timeout2->next-tick2->timeout3->timeout4

### 9.嵌套比较深的数组定义一个方法 展开成一个一维数组
```
    //普通递归方式
    let arr = [1,2,[3,[4,5]]];
    let result = [];
    function flatten(arr){
        for(let i=0;i<arr.length;i++){
            let item = arr[i];
            if(Array.isArray(arr[i])){
                flatten(item)
            }else{
                result.push(item)
            }
        }
        return result
    }
    console.log(flatten(arr))
```
```
    //加强版
    // map 在数组的每一项上调用callback来创建新数组,然后递归
    //concat连接两个数组
    let arr = [1,2,[3,[4,5]]];
    function flatten(arr){
        return [].concat(...arr.map(item=>Array.isArray(item)?flatten(item):item))
    }
    console.log(flatten(arr))
```
